# 講師向け授業通知 設計書

作成日: 2026-07-11
対象システム: 武蔵野個別指導塾 授業スケジューラ（sync-app.html + Google Apps Script）
対象読者: 塾長（意思決定者）／実装担当者

---

## 0. この文書の目的

現在のスケジューラは、塾長・事務局が単一のHTML画面（sync-app.html）を開き、Google Apps Script（GAS）の `saveState` / `getState` を通じて授業データ（講師・生徒・授業・週）をやり取りする仕組みです。

ここに新しく「講師ごとに、自分の授業予定を能動的に知らせる」通知の仕組みを追加したいという要望が出ています。本書は、その実現方法を比較検討し、塾長が意思決定できる材料をまとめたものです。

**本書はコードを変更していません。設計案の提示のみです。**

---

## 1. 目的と前提

### 1.1 なぜ「画面共有」ではなく「プッシュ通知」なのか

現行のスケジューラ画面は、次のような理由から講師に直接共有する運用に向いていません。

- 画面には全講師・全生徒の授業データが一覧表示される。講師Aに見せると、講師B〜Rの授業や、全生徒の氏名・学年・学校名まで見えてしまう。
- 編集用の画面であり、誤操作で他の授業を書き換えてしまうリスクがある（閲覧専用モードは現状ない）。
- 講師は「自分が何日にどの生徒の授業を担当するか」だけ知りたいのであって、全体の時間割を読み解く必要はない。

したがって、**講師には「自分の予定」だけを、こちらから知らせに行く（プッシュする）**方式が望ましい、という前提で以下を検討します。

### 1.2 個人情報の観点（生徒名をどこまで載せるか）

通知文に生徒の実名を含めるかどうかは、業務上の利便性と個人情報保護のトレードオフです。

- **載せるメリット**: 講師が事前に生徒名で教材・進捗を確認できる。連絡ミスが減る。
- **載せないメリット**: 万一スマホの通知プレビュー（ロック画面）を第三者に見られても、生徒の氏名が漏れない。LINEやメールは家族と共有端末で使っているケースもあり、ロック画面通知に個人情報が出ることを嫌う塾もある。
- 折衷案として、**「生徒イニシャル」または「生徒ID・コマ番号」表記**にして、詳細は講師専用ページ（案D）で確認してもらう方法もある。

→ **この「生徒名の掲載可否」は本書では結論を出さず、塾長の判断事項として4章末に残します。** 以降の文面例では「載せる版」「伏せる版」の両方を用意します。

### 1.3 データ面の前提（現状のスケジューラ構造）

`sync-app.html` 内の状態オブジェクト（GAS側に保存される）は以下の構造を持っています。実装時はこれをGAS側から読めば講師別の予定が抽出できます。

```
teachers: [{ id, name, ngDays, note, subjects }]   // 講師マスタ（約17〜18名）
students: [{ id, name, grade, school, note, prefs }]
weeks:    [{ id, name, startDate }]                 // 週の月曜日の日付
lessons:  [{ id, weekId, day(0=月〜6=日), slotId, teacherId, studentId, subject, status, memo }]
slots:    [{ id, label }]                           // 例: sl3 = "16:40-18:10"
```

つまり、ある講師の「明日の授業」は `lessons` を `teacherId` と `day`（曜日）＋対象週の `startDate` で絞り込めば機械的に抽出できます。この抽出ロジックはA〜C案すべてで共通して使います。

**なお、現状リポジトリにあるのはこの構造の「初期サンプルデータ」であり、実際に運用中の全講師・全授業データは本番のGASプロパティ（サーバー側）に保存されています。** 後述する配信通数の試算は、実データで検証する必要があります。

### 案A: LINE公式アカウント + Messaging API プッシュ配信

塾のLINE公式アカウントを開設し、GASからMessaging APIでpush送信する方式です。塾長の第一希望。

**重要な注意**: 以前よく使われていた「LINE Notify」は **2025年3月31日にサービス終了済み**で、現在は新規発行・利用ともにできません。したがって実装するなら最初から「LINE公式アカウント＋Messaging API」一択になります（LINE Notifyを使った簡易実装の記事がネット上に多く残っていますが、それらはすべて使えません）。

**費用試算（要確認・2026年7月時点の一般公開情報に基づく）**

LINE公式アカウントの料金プランは次の3段階です（2026年7月時点。Web検索で確認、ただしLINE社は改定を行うことがあるため実装直前に公式サイトで再確認してください）。

| プラン | 月額固定費 | 無料メッセージ通数/月 | 追加メッセージ |
|---|---|---|---|
| コミュニケーションプラン | 0円 | 200通 | 不可 |
| ライトプラン | 5,000円（税別。税込み目安 約5,500円） | 5,000通 | 不可 |
| スタンダードプラン | 15,000円（税別） | 30,000通 | 従量課金で追加可能 |

（出典: LINE Developers公式ドキュメント「Messaging APIの料金」および複数の代理店解説記事。2026年10月に追加メッセージ従量課金の料金体系見直しが予定されているとの情報もあり、コミュニケーション/ライトプランの無料通数自体への影響は現時点で確認されていませんが、契約直前に公式情報を再確認することを推奨します。）

通数のカウント方法は「送信先ユーザー数 × メッセージ数（吹き出し数）」です。1人の講師に1通（1吹き出し）で当日分をまとめて送れば、「その日に通知した講師の人数」がそのまま通数になります。

- 講師は約17名。仮に「毎日全員に送る」場合、17通/日 × 30日 = 510通/月 → **無料の200通を大きく超過します。**
- 要望どおり「翌日に授業が入っている講師だけに、前日20時に配信」する場合、1日あたりの対象講師数が鍵になります。仮に1日あたり平均10〜15名程度が授業を持つと想定すると、
  - 10名/日 × 26稼働日/月 ≈ 260通/月
  - 15名/日 × 26稼働日/月 ≈ 390通/月
  - いずれのケースでも **無料枠200通/月は超過する可能性が高い** です（対象講師が1日あたり8名未満に収まる場合のみ200通以内に収まりますが、17名体制でこの想定は楽観的すぎます）。
  - ※ この人数は本リポジトリのサンプルデータからの概算であり、実際の稼働講師数・週あたり営業日数で変わります。**本番導入前に、直近1か月の実データから「日別の稼働講師数」を集計して検証することを強く推奨します。**
- 上記の試算から、**実運用では無料のコミュニケーションプランでは通数が不足し、ライトプラン（月5,000通・月額5,000円＋税）への切り替えがほぼ必須**と考えられます。ライトプランなら5,000通/月の枠があり、17名×毎日配信（510通/月）は問題なく収まります。

### 案B: メール配信（GAS標準機能）

GASの `MailApp` または `GmailApp` を使い、講師のメールアドレス宛に翌日の予定を送るだけのシンプルな方式です。

- 費用は0円。実装もGASにコードを数十行追加するだけで、**最短で当日〜翌日には稼働開始できます。**
- GASの送信上限（Googleの割り当て）は、無料のGoogleアカウント（個人のGmail）で**1日あたり100件（送信先アドレス単位でカウント）**、Google Workspaceアカウントであれば1日あたり1,500件です（要確認: Googleの公式クォータページに基づく、変更の可能性あり）。講師17名への送信であれば、無料アカウントの上限（100件/日）でも十分足ります。
- 弱点は「メールは見落とされやすい」こと。ただし件名に「【明日の授業予定】」のように入れておく、送信時刻を講師の生活リズムに合わせる（例: 前日20時）などの工夫で実用に耐えるケースは多いです。

### 案C: Googleカレンダー連携（GASの CalendarApp）

GASの `CalendarApp` を使い、講師ごとの予定として自動でカレンダーに授業を登録（または招待）する方式です。

- 費用は0円。
- 最大の利点は、**講師のスマホにカレンダーアプリの標準通知（30分前・当日朝など）が自動で付く**ことです。プッシュ通知としての即時性は案Aに近く、講師側の追加操作もほぼ不要です。
- 前提として講師がGoogleアカウントを持っている必要があります（17名全員が持っているか要確認）。共有カレンダーへの招待だと「参加を承諾」してもらう一手間が発生する点に注意。
- 授業の変更・振替が起きた場合の「既存カレンダーイベントの更新・削除」ロジックが必要になり、その分だけ案Bより実装がやや複雑になります（授業ID⇔カレンダーイベントIDの対応表をGAS側で持つ設計が必要）。

### 案D: 講師専用の読み取り専用ページ（トークン付きURL）

GASで講師ごとに固有のトークンを発行し、`https://.../exec?token=xxxx` のようなURLにアクセスすると、その講師の予定だけを表示する簡易ページを返す方式です。

- これ自体はプッシュ通知ではありません。**「通知は来るが詳細を見たい」というときの受け皿**として、案A〜Cと組み合わせるのに向いています（例: LINEやメールの本文には要点だけ書き、「詳しくはこちら」というリンクを添える）。
- 生徒名などの個人情報をどこまで載せるかも、通知本文より柔軟に調整できます（ページ側でログイン相当のトークン確認をしているため、通知本文より安全に詳細を出せる）。
- 実装は小さめですが、**トークンの管理（漏洩・失効の運用）を誰が責任者としてやるか**を決めておく必要があります。

## 3. 推奨案

**結論: 「案B（メール）または案C（カレンダー）で即日〜数日内に運用を開始し、並行して本命の案A（LINE）を準備・移行する」段階導入を推奨します。**

理由:

1. 塾長の第一希望は案A（LINE）ですが、LINE公式アカウントの開設・Messaging APIチャネルの発行・講師のuserId収集（友だち追加＋合言葉運用）には、塾側・講師側の両方に一定の準備期間が必要です。この間、講師への通知が「ゼロ」の状態が続くのは避けたい。
2. 案B（メール）は実装がごく小さく、**既存の講師メールアドレスがあれば当日中にも稼働できる**ため、まず最短ルートで「授業がある日は通知が来る」体験を先に用意できます。
3. 案C（カレンダー）は案Bよりややプッシュ性が高く（通知が端末に出る）、講師が全員Googleアカウントを使える前提が確認できれば、Bの代わり、またはBと同時に用意してもよいでしょう。
4. 案Aは費用（月200通では不足しライトプラン必須の可能性が高い）と初期設定の手間があるため、"本命"として並行準備し、講師のuserId登録が一定数集まった段階で本番切り替えるのが無理がありません。
5. 案Dは通知そのものではないため優先度は低いですが、A・Cのメッセージ本文を短くしたい場合の補完として、実装コストが低い範囲で後回しにしても構いません。

**代替判断もありえます**: もし塾長が「多少の初期投資と数日の準備期間をかけてでも、最初からLINEだけに一本化したい」という場合は、案Bを飛ばして案Aに直接着手する判断も合理的です。その場合は4章の手順に沿って進めてください。

## 4. 案A（LINE）の実装設計詳細

### 4.1 導入手順の概要

1. **LINE公式アカウントを開設**（LINE for Business の管理画面から。既に塾のアカウントがあればそれを流用可）。
2. LINE Developersコンソールで、そのLINE公式アカウントに紐づく **Messaging APIチャネル** を作成。
3. チャネル基本設定画面で **チャネルアクセストークン（長期）** を発行し、GAS側のスクリプトプロパティ（`PropertiesService`）に保管する。
4. Webhook URLに、後述するGASの `doPost` 関数（Webアプリとしてデプロイしたもの）のURLを設定し、Webhookを有効化する。
5. 料金プランを選択（3章・2章参照。当面はコミュニケーションプランで様子を見て、通数超過が見えた時点でライトプランへ変更する運用でも可）。

### 4.2 講師のuserId収集方法

LINEのpush送信には、講師個人を識別する `userId`（LINE内部ID。電話番号やLINE IDとは別物）が必要です。これを次の手順で集めます。

1. 講師に、塾のLINE公式アカウントを**友だち追加**してもらう（QRコード・招待リンクを配布）。
2. 友だち追加後、講師に**合言葉（例: 自分の講師コードや氏名など、事前に個別配布した一意な文字列）をトーク画面に送ってもらう**よう案内する。
3. GAS側の `doPost`（Webhook受信口）で、届いたメッセージの `event.source.userId` と、本文に書かれた合言葉を受け取る。
4. 合言葉が講師マスタ（`teachers`）内の既存の講師と一致すれば、その講師レコードに `lineUserId` を紐付けて保存する。一致しない・不正な合言葉の場合は、講師に「合言葉が確認できませんでした」等の返信メッセージを返す。

この方式であれば、講師が電話番号や個人情報をやり取りする必要がなく、事務局側も「誰が友だち追加したか」を自動で講師マスタに反映できます。

### 4.3 配信ロジック

- GASの**時間主導トリガー**を設定し、**毎日20:00**に配信バッチ関数を実行する。
- バッチ処理の流れ:
  1. 「翌日の日付」に対応する週（`weeks`）と曜日（`day`）を特定する。
  2. その `day` に紐づく `lessons` を全件抽出する。
  3. 抽出した授業を `teacherId` ごとにグループ化する。
  4. 各講師について、`lineUserId` が登録済み・かつ通知ON（`notifyEnabled`）であれば、その講師の翌日分の授業をまとめた1通のメッセージを作成し、Messaging APIのpushエンドポイントに送信する。
  5. 授業がない講師には何も送らない（＝通数を無駄に消費しない）。

### 4.4 メッセージ文面の例

**① 生徒名を載せる版（塾長が「載せてよい」と判断した場合）**

```
【武蔵野個別指導塾】明日 7/12(日) の授業予定

16:40-18:10 生徒: 山田太郎さん（数学）
18:20-19:50 生徒: 佐藤花子さん（英語）

よろしくお願いいたします。
```

**② 生徒名を伏せる版（個人情報配慮を優先する場合）**

```
【武蔵野個別指導塾】明日 7/12(日) の授業予定

16:40-18:10 数学（1コマ）
18:20-19:50 英語（1コマ）

詳細はスケジューラ担当までご確認ください。
```

※ ②の場合、詳細確認の手段として案D（トークン付き閲覧ページ）を併用すると、講師の利便性を落とさずに通知本文の個人情報を減らせます。

### 4.5 スケジューラ側に必要な変更

現状の講師マスタ（`teachers`）に、以下のフィールドを追加する程度の変更で足ります（本書は設計のみで、実装はしていません）。

- `lineUserId`: string（未登録なら空文字）
- `notifyEnabled`: boolean（講師ごとに通知のON/OFFを切替可能にする。学生アルバイト講師で通知不要な人がいる場合などに対応）

画面側（sync-app.html）では、講師マスタ編集画面に「LINE連携状況（未連携／連携済み）」の表示と、通知ON/OFFのチェックボックスを追加する程度で十分です。

### 4.6 GAS側の擬似コード（実装イメージ）

```javascript
// --- Webhook受信: 講師のuserIdを紐付ける ---
function doPost(e) {
  const event = JSON.parse(e.postData.contents).events[0];
  if (event.type === 'message' && event.message.type === 'text') {
    const userId = event.source.userId;
    const text = event.message.text.trim();
    const teacher = findTeacherByPassphrase(text); // 講師マスタから合言葉一致を検索
    if (teacher) {
      teacher.lineUserId = userId;
      teacher.notifyEnabled = true;
      saveTeachers(); // 状態を保存
      replyLine(event.replyToken, '連携が完了しました。今後、授業予定をお送りします。');
    } else {
      replyLine(event.replyToken, '合言葉が確認できませんでした。事務局にご確認ください。');
    }
  }
}

// --- 時間主導トリガー: 毎日20:00に実行 ---
function sendTomorrowNotifications() {
  const tomorrow = addDays(today(), 1);
  const { week, day } = resolveWeekAndDay(tomorrow);
  const lessons = getLessonsFor(week.id, day);

  const byTeacher = groupBy(lessons, l => l.teacherId);

  for (const teacherId in byTeacher) {
    const teacher = getTeacherById(teacherId);
    if (!teacher.lineUserId || !teacher.notifyEnabled) continue;

    const message = buildMessage(teacher, byTeacher[teacherId], tomorrow);
    pushLineMessage(teacher.lineUserId, message);
  }
}

function pushLineMessage(userId, text) {
  UrlFetchApp.fetch('https://api.line.me/v2/bot/message/push', {
    method: 'post',
    headers: { Authorization: 'Bearer ' + CHANNEL_ACCESS_TOKEN },
    contentType: 'application/json',
    payload: JSON.stringify({ to: userId, messages: [{ type: 'text', text }] })
  });
}
```
