# 授業スケジューラ 改修プロンプト集（2026-07-11）

Claude Code に投げるための要件定義済みプロンプト3本。実装担当（Sonnet）にそのまま渡せる粒度で記述してある。
※ 本リポジトリでは3本とも実行・反映済み。再利用や修正指示のたたき台として保存する。

## 前提（3本共通で冒頭に付ける文脈）

- 対象は単一ファイルWebアプリ `sync-app.html`（GAS同期版・素のJS・フレームワークなし）。`index.html` は旧・単体版で構造が異なる。`juku-github更新/sync-app.html` はルート版と同一内容を維持するコピー。
- 状態オブジェクト `S`：`teachers` / `students` / `lessons`（`weekId`・`day`(0=月〜6=日)・`slotId`・`teacherId`・`studentId`）/ `weeks` / `slots` / `ui`（表示中の週 `S.ui.weekId`・曜日 `S.ui.day`）。
- 生徒の同定は名前の完全一致文字列。講師のその日のNG判定は `isNg(weekId,day,teacherId)`、臨時稼働は `isRinji(...)`。
- コードスタイルは既存に合わせる（圧縮気味のJS・日本語コメント・既存CSS変数）。ライブラリ追加禁止。コミットは監督者が行う。

## プロンプト1：日別の講師優先表示

> 時間割グリッドで日タブを切り替えたとき、その日に関係の深い講師が上に並ぶようにする。
>
> 1. 新関数 `dayTeacherOrder(weekId, day)` を追加。優先度は上から：
>    - 第0キー: `sortPin==='top'` は常に最上位、`'bottom'` は常に最下位（既存仕様の尊重）
>    - 第1キー: その週・その曜日に授業が1件以上ある講師が上
>    - 第2キー: 授業が無い講師同士では、その日に稼働可能（`!isNg` または `isRinji`）な講師がNG講師より上
>    - 第3キー: その日の授業数降順
>    - 第4キー: 既存の総授業数降順 → 名前順（`localeCompare(…,'ja')`）
> 2. `renderGrid()` の行生成 `const rows=teacherOrder().filter(...)` だけを `dayTeacherOrder(w.id,day)` に差し替え。inactiveフィルタ条件は維持。
> 3. その日に授業がある講師の名前セル（`th.tname`）に控えめな目印（既存CSS変数を使った左ボーダー程度）を付ける。
> 4. 他画面（マスタ一覧・帳票・授業ダイアログの講師セレクト等）の `teacherOrder()` 呼び出しは変更しない。影響範囲はグリッド表示のみ。
>
> 検証：`<script>` 内JSを抽出して構文チェック。ルートと `juku-github更新/` の sync-app.html が diff で一致すること。

## プロンプト2：講師への授業通知（設計書の作成）

> コードは変更せず、`docs/講師通知設計.md` を日本語で新規作成。塾長（非エンジニア）が読んで意思決定できる文書にする。
>
> 背景：バックエンドはGASなので、サーバー側で全授業データにアクセスでき、毎日定時実行のトリガーも使える。要望は「画面共有ではなく、各講師に授業日をプッシュで伝えたい。第一希望はLINE」。講師は約17名。
>
> 構成：
> 1. 目的と前提（画面共有をやめる理由、生徒名をどこまで載せるかの個人情報論点）
> 2. 案の比較（表＋説明）：
>    - 案A: LINE公式アカウント + Messaging API プッシュ（旧LINE Notifyは2025年3月末終了済みと明記。無料枠 月200通の超過試算とライトプランの費用を提示。料金は最新情報を確認し、確認できなければ「要確認」注記）
>    - 案B: メール配信（GAS MailApp/GmailApp。無料・実装最小・送信上限に言及）
>    - 案C: Googleカレンダー連携（CalendarApp で講師のカレンダーに自動登録。スマホ通知が自動で付く）
>    - 案D: 講師ごとのトークン付き閲覧専用ページ（プッシュではない補完策）
>    - 比較軸：初期設定の手間（塾側・講師側）／月額費用／確実に届くか／実装工数／運用の手間
> 3. 推奨案（段階導入を基本に、分析次第で変更可。理由明記）
> 4. 案Aの実装設計詳細（公式アカウント開設→チャネル作成→トークン取得の手順、講師userIdの収集フロー＝友だち追加+合言葉+doPost Webhook、毎日20時に翌日分を集計し授業がある講師にのみpush、生徒名を載せる版/伏せる版の文面例、講師マスタへの `lineUserId`・通知ON/OFF欄追加、GAS擬似コード）
> 5. 塾長が決める次のアクション（案の選択・生徒名掲載可否・配信タイミング）
>
> 注意：自信のない料金・仕様は断定せず「要確認」を付す。

## プロンプト3：生徒の名寄せ（表記ゆれ重複の検出・統合）

> 背景：Excel取込データで同一人物が「生沼はな／生沼映南」「陳彦浩／陳やすひろ」のように漢字・かな表記ゆれで二重登録されている疑い。生徒同定が名前の完全一致のため、1文字違えば別人扱いになる。生徒IDを参照するのは実質 `lessons` のみ。
>
> 3-1. 候補検出 `dupCandidates()`（自動統合はしない。候補提示のみ）：
> - 正規化一致（NFKC＋全半角スペース除去）
> - 姓一致（先頭2文字一致かつ残りが異なる）
> - 先頭1文字一致＋名の文字種ミスマッチ（片方かな主体・片方漢字主体。「陳彦浩／陳やすひろ」対策）
> - ペアは重複なく名前順で安定的に列挙
>
> 3-2. マスタ画面に「⚠ 同一人物の可能性がある生徒」ボックスを追加。各ペアに名前・学年・学校・授業数を併記し「統合…」ボタンを置く。候補0件なら控えめに表示。
>
> 3-3. 統合処理 `mergeStudents(keepId, removeId)`：
> - `lessons.studentId` を全件付け替え／`prefs` は和集合／連絡先系フィールドは空欄補完／`note` は連結し「（旧表記: ○○）」を必ず追記／消す側レコードを削除／`save()`＋再描画＋トースト通知
> - 確認ダイアログに「統合すると元に戻せません」を明記。残す名前を選ばせる。
>
> 3-4. 授業登録時の予防：`updateStudentInfo()` で未登録名の入力時、似た既存生徒がいれば「もしかして: 生沼はな（中2）？」を表示し、クリックで入力欄に反映。
>
> 検証：構文チェックに加え、検出ロジックを node で切り出し「生沼はな/生沼映南」「陳彦浩/陳やすひろ」「田中太郎/田中太郎」が検出され「佐藤一/鈴木二」が検出されないことを確認して報告する。
